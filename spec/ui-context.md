# Chess Analysis App — UI-контекст

## UI-ціль

Застосунок має відчуватись як сфокусований робочий простір для шахового аналізу, а не як маркетинговий лендінг. Перший корисний екран після входу — dashboard, де користувач може імпортувати партії і переглянути останні результати.

## Дизайн-референс

Для v1 є окремий дизайн-прототип у `design-prototype/`. Детальний опис прийнятих і відхилених рішень лежить у `design-prototype.md`.

Прототип використовується як візуальний референс, але не як production-код. Реалізація має залишатися в межах Next.js + TypeScript + CSS Modules і не копіювати inline styles або CDN/Babel setup із прототипу.

## Мова

- UI-мова для v1: українська.
- Мова LLM-аналізу для v1: українська.
- Шахова нотація може лишатися стандартною PGN / SAN.
- Технічні назви Chess.com, Lichess, Stockfish і GitHub не перекладаються.

## Тон продукту

- Прямий, практичний, спокійний.
- Орієнтований на coaching, але без зайвої багатослівності.
- Вважати, що користувач має приблизно 1000 ELO.
- Надавати перевагу конкретним наступним діям, а не абстрактній шаховій теорії.

## Візуальний напрям

- Темна шахова workspace-естетика.
- Основна палітра може стартувати з темно-зеленої теми прототипу.
- Дошка і аналітика мають бути головними візуальними об'єктами.
- Desktop navigation може бути вузьким sidebar з іконками.
- Mobile navigation може бути bottom nav.
- Числа, rating, evaluation і короткі stats можна набирати моноширинним шрифтом.

## Принципи макета

- Пріоритет: scanability і зручність для повторного використання.
- Уникати декоративних landing-page секцій після авторизації.
- Використовувати щільні, але читабельні dashboards і tables.
- На сторінці партії дошка і список ходів мають бути візуально центральними.
- Не вкладати cards всередину cards.
- Не використовувати oversized hero typography всередині app panels.
- Використовувати стабільні розміри для chessboard, eval bar, graph, tables, filters і action buttons.

## Основні екрани

### Стартова сторінка

Коли користувач не авторизований:

- показує назву продукту;
- пропонує GitHub login;
- не показує import або analysis controls.

Коли користувач авторизований:

- редіректить або чітко веде на `/dashboard`.

### Сторінка входу

- Одна основна дія: продовжити з GitHub.
- Без email/password UI у v1.

### Дашборд

Призначення:

- імпортувати партії;
- бачити список імпортованих партій;
- фільтрувати і відкривати партії;
- вибирати партії для групового аналізу в пізніших фазах.

Елементи керування:

- вибір платформи: Chess.com / Lichess;
- поле для нікнейму;
- сегментований перемикач кількості партій: 25 / 50 / 100;
- сегментований перемикач періоду: 7 / 30 / 90 днів;
- кнопка імпорту;
- фільтри платформи, контролю часу, результату і діапазону дат.

Рядок партії має показувати:

- суперника;
- результат;
- колір гравця;
- дебют;
- контроль часу;
- дату партії;
- статус engine-аналізу;
- статус LLM-аналізу.

### Сторінка партії

Призначення:

- відтворення однієї партії;
- перегляд висновків Stockfish;
- запуск і читання LLM-аналізу.

Основні зони:

- шахова дошка;
- керування ходами;
- PGN move list;
- eval bar;
- eval graph;
- стрілка найкращого ходу або підсвічування;
- класифікація ходів;
- панель LLM-аналізу.

Очікувані елементи керування:

- перший хід;
- попередній хід;
- наступний хід;
- останній хід;
- запустити Stockfish review;
- проаналізувати партію через LLM;
- повторити LLM-аналіз, якщо кешований аналіз уже існує.
- мають малюватися стрілочки найращого ходу
- прямо на дошці показуються іконки якості ходу (помилка, грарно, накращий і тд)


Stockfish review має бути схожим на звичні інструменти chess game review:

- поточна оцінка завжди видима;
- великі помилки легко помітити;
- графік дозволяє переходити до ходу;
- ключові моменти описані простою мовою;
- accuracy видима для обох сторін.

### Сторінка профілю

Призначення:

- показати останню результативність і накопичувальне coaching-зведення.

Елементи керування:

- вибір кількості партій: 25 / 50 / 100;
- вибір періоду: 7 / 30 / 90 днів;
- оновлення LLM-зведення.

Метрики:

- перемоги / нічиї / поразки;
- результативність за контролем часу;
- результативність за кольором;
- найчастіші дебюти;
- win rate за дебютом.

ELO-секція:

- Два окремі графіки: Chess.com і Lichess (рейтинги не порівнянні між платформами).
- Обидва будуються з `games.player_rating`, відсортованих по `played_at`.
- Графік не показується, якщо для платформи немає імпортованих партій.
- Всі інші метрики профілю об'єднують обидві платформи.

## Візуальні компоненти

- Використовувати CSS Modules.
- Використовувати нативні controls там, де їх достатньо.
- Використовувати сегментовані controls для фіксованих варіантів.
- Використовувати чекбокси для вибору партій.
- Використовувати кнопки зі зрозумілими labels або icons, де доречно.
- Використовувати таблиці або компактні списки для партій.
- Використовувати графіки тільки коли вони покращують scanability.

## Порожні стани і стани завантаження

Обов'язкові стани:

- користувач не авторизований;
- немає імпортованих партій;
- імпорт триває;
- імпорт завершився, але партій не знайдено;
- Stockfish-аналіз не запущено;
- Stockfish-аналіз триває;
- LLM-аналіз не запущено;
- LLM-аналіз триває;
- profile має менше ніж 5 партій.

## Адаптивність

- Desktop-версія сторінки партії може використовувати дошку й панелі аналізу поруч.
- Mobile-версія сторінки партії має складати дошку, controls, ходи й аналіз вертикально.
- Chessboard має залишатися квадратною.
- Eval graph має лишатися читабельним або згортатися в компактний вигляд на вузьких екранах.
- Текст в елементах керування не має виходити за межі контейнера.
